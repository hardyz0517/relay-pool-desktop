# Relay Pool Desktop v0.3.2

v0.3.2 upgrades the local data layer to Persistence Architecture V2. The localhost OpenAI-compatible endpoint and normal workflows are unchanged; the main changes are data migration, startup recovery, and persistence reliability.

## Highlights

- Replaces the legacy `AppDatabase` boundary with SQLx SQLite, focused stores, application services, and bounded write coordination.
- Imports a v0.3.1 database read-only into a separate generation-2 database on first start. The source is not overwritten before verification, and a verified backup is retained.
- Uses a durable upgrade journal and deterministic recovery decisions after interrupted upgrades. Unknown or conflicting state fails closed instead of guessing or overwriting data.
- Preserves stations, station keys, credential references, settings, request logs, request-attempt lifecycle fields, and change history. Legacy secrets are re-encrypted with V2 AAD.
- Fixes the startup recovery response field mismatch that could show `invalid data store startup response` even when the database was healthy.

## Before upgrading

- Supported paths are Windows x64 fresh install or direct upgrade from v0.3.1. For v0.3.0 or earlier, first upgrade to v0.3.1, start it successfully once, and fully exit it.
- Fully exit Relay Pool Desktop from the tray. Do not run v0.3.1 and v0.3.2 at the same time.
- Complete any pending data-directory relocation in v0.3.1 and restart before upgrading.
- Back up the data directory only while the app is fully stopped. Database backups contain business data and encrypted credential ciphertext, depend on the original Windows user credential store, and are not portable exports.

## Upgrade and recovery

- The first start creates and validates the V2 database, so it can take longer than a normal start.
- If the recovery screen appears, do not delete, rename, or overwrite database, backup, journal, or tombstone files. Preserve the state and use the app's recovery and redacted diagnostics actions.
- Do not open the upgraded data directory with v0.3.1 after migration completes.

## Downgrade

- Automatic down migration is not supported. Reinstalling v0.3.1 does not convert V2 data back to generation 1.
- If a downgrade is unavoidable, first retain the V2 backup, then restore an app-generated and verified generation-1 backup through a controlled recovery process. Do not manually rename database files.

## Known boundaries

- Direct migration from v0.3.0 or earlier is unsupported.
- Backups depend on the original system credential store and do not guarantee secret recovery under another Windows user or on another device.
- This release remains a technical preview and is primarily qualified on Windows x64.
