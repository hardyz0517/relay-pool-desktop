# Relay Pool Desktop 外观基础统一设计

## 1. 目标

把 Relay Pool Desktop 的外观从“页面各自成型”升级为“有统一壳层、统一布局契约、统一组件语言”的桌面工具界面。

本设计只解决外观基础层，不改业务逻辑，不引入新的产品能力。目标是让后续页面和功能都能直接复用同一套 UI 规则，降低长期维护成本。

## 2. 设计背景

当前项目已经具备可用的页面骨架，但外观层还存在几个典型问题：

- 壳层、页面、卡片、表格的布局语法不完全一致
- 不同页面对间距、边界、标题层级的处理方式不统一
- 某些状态依赖局部写法，未来容易继续分叉
- 侧边栏、页头、内容区、详情区之间还没有形成严格契约

这些问题短期看是“观感不整齐”，长期看会变成“每加一个页面就多一套样式习惯”。

## 3. 设计原则

### 3.1 桌面工具优先

界面应持续保持浅色、克制、紧凑、信息密度高的桌面工具气质，不向营销页、SaaS 后台或装饰性界面靠拢。

### 3.2 结构先于装饰

先统一壳层和布局，再统一视觉细节。不能反过来只做局部美化，否则只会把不一致包起来。

### 3.3 页面只负责内容

页面组件负责展示数据和组织内容，不负责定义新的布局习惯。布局规则应该来自统一模板和原语。

### 3.4 状态统一表达

相同语义的状态必须用同一套视觉语法表达，包括默认、悬停、选中、禁用、加载、成功、警告、错误。

### 3.5 可维护优先

任何新增样式都要问两个问题：

- 它能否复用现有原语？
- 它会不会让下一页也跟着复制一遍？

如果答案不理想，就应该抽到基础层，而不是散落在页面里。

## 4. 当前问题分层

### 4.1 壳层层

壳层是整个应用的第一视觉层，当前需要统一：

- 左侧侧边栏的展开/收起契约
- 顶部条的高度、边界和信息密度
- 主内容区的留白和滚动行为
- 底部状态区的固定几何

### 4.2 页面模板层

当前页面类型较多，但模板边界还不够清楚，需要把页面收敛为几种固定模型：

- 总览型
- 列表/详情型
- 表单设置型
- 表格分析型
- 规则配置型

### 4.3 组件原语层

已有部分基础组件，但仍需要明确它们的职责边界，避免继续长成“某页专用组件”：

- `AppShell`
- `PageScaffold`
- `WorkspaceLayout`
- `SectionCard`
- `MetricCard`
- `StatusBadge`
- `ActivityList`
- `DataTableLite`
- `InspectorPanel`
- `Toolbar`

## 5. 总体架构

### 5.1 全局壳层

全局壳层由 `AppShell` 负责，只做三件事：

1. 提供侧边栏
2. 提供顶部状态条
3. 提供主内容容器

壳层不承载页面特例逻辑，不在这里拼凑具体业务组件。

### 5.2 页面标题层

`PageScaffold` 统一页面标题、描述和右侧操作区。它负责页面的顶部语义层，而不是页面内容本身。

### 5.3 工作区布局层

`WorkspaceLayout` 用于列表/详情等双栏或多栏布局，负责列宽和容器间距，不负责业务信息。

### 5.4 内容原语层

`SectionCard`、`MetricCard`、`StatusBadge`、`DataTableLite` 等组件负责稳定展示某类内容。它们应该足够通用，能在多个页面复用。

## 6. 壳层设计规范

### 6.1 侧边栏

侧边栏应成为稳定的结构容器，必须满足：

- 展开态和收起态保留同一套 DOM 结构
- 收起时只变化宽度和文字可见性，不重新排版节点
- 图标、行高、内边距、交互热区保持稳定
- 底部状态和折叠按钮使用与导航项一致的行语法

侧边栏的视觉重点是“稳定”和“可预测”，不是“展开时很花哨”。

### 6.2 顶部条

顶部条必须固定高度，包含：

- 当前页面标题
- 简短描述
- 少量全局状态
- 主要全局动作

顶部条不能因为内容变化而改变整体高度，也不能根据页面临时塞入不同排版。

### 6.3 底部状态区

底部状态区用于表达本地代理、连接状态、活动态势等轻量信息。它必须始终占位，不能在折叠时删除节点或改变布局流，否则会造成视觉抖动。

### 6.4 主内容区

主内容区负责页面滚动，不承担额外壳层样式。页面内部如需滚动，应优先使用统一容器规则，而不是每页自定义滚动语法。

## 7. 页面模板设计

### 7.1 总览型页面

用于展示关键指标、近期活动、状态汇总和快捷入口。

结构建议：

- 顶部指标带
- 左侧主活动区
- 右侧状态汇总区

适用页面：

- `DashboardPage`

### 7.2 列表/详情型页面

用于一侧列表、一侧详情的管理界面。

结构建议：

- 左侧列表，负责扫描和选择
- 右侧详情，负责解释和编辑

适用页面：

- `StationsPage`
- `KeyPoolPage`
- 部分 collectors / channels 类页面

### 7.3 表单设置型页面

用于设置项、偏好项、配置项。

结构建议：

- 分组表单
- 说明文本
- 次级操作区

适用页面：

- `SettingsPage`
- `RoutingPage` 的部分配置区

### 7.4 表格分析型页面

用于价格、日志、状态明细等高密度信息视图。

结构建议：

- 表格主体
- 行级状态
- 右侧详情面板或抽屉

适用页面：

- `PricingPage`
- `LogsPage`

### 7.5 规则配置型页面

用于路由策略、健康策略、过滤规则等配置。

结构建议：

- 规则块
- 当前生效状态
- 结果预览或影响说明

适用页面：

- `RoutingPage`

## 8. 组件原语设计

### 8.1 标题与描述

页面标题、卡片标题、分组标题必须区分层级：

- 页面标题：主级
- 卡片标题：次级
- 辅助说明：低对比度、小字号

禁止同一页面里所有标题都采用同一种重量和字号。

### 8.2 卡片

卡片用于承载单个独立信息块，边界要清晰，圆角和阴影都应克制。

卡片只适合：

- 单个指标
- 单个状态块
- 单个说明区
- 单个编辑区

不应把一整个页面套进多个嵌套卡片里。

### 8.3 表格

表格组件应统一：

- 行高
- 悬停反馈
- 选中态
- 空态
- 排序提示

所有表格页面都要遵守同一套密度规则。

### 8.4 状态徽标

状态徽标必须统一语义映射：

- healthy
- warning
- error
- disabled

禁止每个页面给同一类状态发明新的颜色或说法。

### 8.5 表单控件

输入框、选择器、切换、按钮必须保持统一的高度、边框、焦点态和禁用态。

## 9. 视觉令牌

视觉层要尽量通过 token 驱动，而不是在页面内散写数值。

需要统一的 token 类型包括：

- 背景色
- 前景色
- 边框色
- 语义色
- 圆角
- 阴影
- 字体层级
- 间距档位
- 行高档位

原则是：同一种语义只保留一套默认值，局部例外越少越好。

## 10. 状态系统

所有基础组件应支持以下状态：

- 默认
- 悬停
- 选中
- 禁用
- 加载
- 成功
- 警告
- 错误

这些状态必须跨组件一致，不能出现“按钮一套、表格一套、徽标又一套”的情况。

## 11. 信息密度与阅读节奏

Relay Pool Desktop 是管理工具，不是展示页，因此信息密度应保持高，但阅读节奏要清楚：

- 关键数据优先
- 说明文字次级
- 操作入口固定位置
- 详情解释集中放置

同一屏中只能有一个主阅读方向，其他信息都应辅助它。

## 12. 维护约束

### 12.1 禁止行为

- 不允许页面内随意发明新布局契约
- 不允许同类状态使用不同颜色体系
- 不允许为了单页效果引入一套新的壳层写法
- 不允许用局部微调替代统一 token

### 12.2 推荐行为

- 优先复用现有原语
- 优先抽象重复的结构
- 优先把样式变化收敛到少数基础组件
- 优先让新增页面“套模板”而不是“重做一遍”

## 13. 迁移顺序

### 第一步：壳层统一

先收紧 `AppShell`、`PageScaffold`、`WorkspaceLayout` 的职责边界，确保全局结构稳定。

### 第二步：原语统一

将卡片、徽标、标题、列表、表格、按钮等基础组件的外观和状态统一到一个标准。

### 第三步：页面模板统一

把总览、列表/详情、表单、表格分析、规则配置这些页面类型固定下来。

### 第四步：视觉抛光

最后再处理细节差异，如间距、边框、圆角、字体层级、hover 节奏。

## 14. 验收标准

以下条件同时满足时，视为外观基础统一完成：

- 侧边栏展开/收起无抖动、无跳变
- 顶部条和底部状态区保持固定几何
- 同类页面共享同一类布局模板
- 卡片、表格、徽标、按钮的视觉语言统一
- 新页面无需重新设计骨架即可接入
- 后续风格调整主要改 token 和基础组件，而不是逐页改样式

## 15. 非目标

本次设计不包含：

- 新的业务功能
- 深色主题主线重做
- 动画效果堆叠
- 仪表盘营销化
- 企业后台模板化
- 复杂交互重构

## 16. 风险与约束

- 如果只改表层不统一骨架，后续仍会回到碎片化
- 如果基础组件职责划分不清，页面会重新长出特例
- 如果 token 不收口，细节会继续分叉
- 如果迁移顺序颠倒，维护成本会先升后降，体验不稳定
