# Relay Pool Desktop UI 全面升级设计方案

日期：2026-06-28

## 背景

当前 Relay Pool Desktop 的界面已经完成了基础可用骨架，但整体仍偏“后台仪表盘”：侧栏较重、首页指标卡分散、对象操作入口不够自然、添加 Provider 仍缺少沉浸式流程。后续 UI 升级应学习 cc-switch 的桌面工具感，但不照搬其业务结构或代码实现。

参考对象：

- cc-switch 主界面：白底、宽松对象列表、顶部轻量工具区、当前使用状态清晰。
- cc-switch 添加 Provider：页面级添加流程，先选预设，再补必要字段。
- cc-switch 组件语义：shadcn/Radix 风格基础组件、按钮层级明确、对象行承载操作。

## 设计目标

1. 以白底、系统灰、Apple 蓝为主视觉基调。
2. 保留侧栏，但弱化成窄工具栏，避免传统后台感。
3. 首页保留重要指标，并让指标比普通状态条更突出。
4. 页面主角从“指标卡片墙”转向“可操作对象”：Provider、中转站、Key、路由队列。
5. 添加 Provider 改为沉浸式页面流程，优先选择 preset。
6. 建立可维护的组件语义：Button、Card、Tabs、Dialog、ObjectRow、MetricPanel 各司其职。

## 视觉原则

基础色：

- 背景：纯白 `#ffffff`。
- 文字：系统黑灰，正文接近 `#111827`，次级信息接近 `#6b7280`。
- 边框：中性浅灰，避免青色边框成为默认视觉。
- 主动作：Apple 蓝，接近 `#0A84FF`。
- 状态色：绿色、橙色、红色只表达运行、告警、错误，不作为大面积主题色。

阴影与圆角：

- 主对象行和首页核心区可以有轻阴影，表达浮起和可操作。
- 普通容器和卡片默认使用 8px 圆角，不使用过度装饰。
- 减少卡片套卡片。页面本身不是一张大卡片，卡片只用于指标块、对象行、摘要和表单分区。

## 全局壳层

采用“S1：纯图标窄工具栏 + 页头说明”。

布局：

- 左侧固定 58-64px 窄工具栏。
- 侧栏只承载一级导航图标和设置入口。
- 当前页面用深灰或黑色底高亮，不使用青色。
- 每个图标必须有 tooltip、`aria-label` 和 title。
- 页面标题、页面说明、状态胶囊、主动作放在顶部 header。

导航语义：

- 侧栏负责一级页面切换：工作台、中转站、Key 池、采集、路由、日志、设置等。
- 页面内部的局部切换使用 Tabs/Segmented Control，不挤进全局导航。
- 保留折叠/展开不是核心目标，因为默认已经是窄栏；如果实现，只作为辅助。

## 首页：代理工作台

采用“H3 的上半区 + H1 的轻对象列表”。

首页首屏结构：

1. 顶部 header：
   - 页面标题：代理工作台。
   - 说明：当前路由、今日指标、轻对象队列。
   - 右侧：本地代理入口、复制入口、添加 Provider。

2. 双核心区：
   - 左侧：当前路由。
     - 当前 Provider 名称、站点图标、base URL。
     - 运行状态、路由策略、代理状态。
     - 主要操作：切换路由、复制入口、打开详情。
   - 右侧：今日指标。
     - 今日请求。
     - 可用 Key。
     - 失败率。
     - 今日成本。

3. 轻对象队列：
   - 显示当前路由队列或关键 Provider/Key。
   - 行内展示优先级、当前使用、余额、健康状态。
   - 次要操作默认收起，hover/focus 时出现。

指标原则：

- 首页指标要突出，但不拆成一整屏指标卡。
- 指标要和“当前路由”同级出现，服务运行判断。
- 中转站和 Key 池页面可以把指标放入对象行，首页不需要每行都密集展示所有指标。

## 对象列表

Provider、中转站、Key、采集任务、路由队列统一采用 ObjectRow 语法。

ObjectRow 基本结构：

- 拖拽柄，存在于可排序列表。
- 图标或缩写。
- 主标题：Provider / 站点 / Key 名称。
- 副标题：base URL、模型、分组、备注等。
- 状态徽标：当前使用、运行中、低余额、禁用、失败。
- 右侧关键指标：请求数、失败率、延迟、余额、优先级。
- hover/focus 动作区：编辑、复制、测试、启用、删除等。

交互规则：

- 对象行默认安静，避免每行铺满按钮。
- 危险操作不能只靠 hover 暴露，必须有确认。
- 当前使用对象应有持久视觉：边框、轻背景或状态徽标。
- 拖拽排序只在可排序列表出现，不要所有列表都显示拖拽柄。

## 添加 Provider

采用“P1：单页预设优先”的沉浸式页面。

入口：

- 从工作台和中转站页的主按钮进入。
- 进入后替换主内容区域，不作为小弹窗。
- 顶部有返回按钮、标题和简短说明。

流程：

1. Provider Preset 区：
   - Custom。
   - OpenAI Compatible。
   - 常用中转站 preset，例如 PackyCode、DeepSeek、Qwen、MiniMax、SiliconFlow。
   - Sub2API 和 NewAPI 可作为特殊 preset。

2. 自动填充：
   - 图标。
   - Provider 名称。
   - Base URL。
   - 站点类型。

3. 用户填写：
   - API Key。
   - 备注。
   - 可选采集策略。
   - 可选健康检测。

4. 底部固定动作栏：
   - 取消。
   - 测试连接。
   - 添加。

特殊情况：

- Sub2API/NewAPI 可以在同一页面展开局部二段配置。
- 不把所有添加都做成向导，避免普通 Provider 添加变慢。
- 复杂编辑同样使用页面级表单，而不是拥挤 Dialog。

## 基础组件语义

Button：

- `primary`：每页最多一个，表示添加、保存、启动等主动作。
- `secondary`：普通次动作，如复制、测试、导入。
- `ghost`：导航、轻操作和工具栏按钮。
- `icon`：只有图标的工具按钮，必须有 tooltip 和 `aria-label`。
- `danger`：删除、清空、重置等危险动作。

Card：

- 用于单个指标块、对象行容器、摘要面板和表单分区。
- 不用于把整个页面包起来。
- 避免卡片内再套视觉上同级的卡片。

Tabs / Segmented Control：

- 只做页面内部视图切换。
- 不承担全局导航。
- 高度保持紧凑，默认灰底，选中项白底轻阴影。

Dialog：

- 用于确认、轻量设置、危险动作提示。
- 不承载复杂添加 Provider 或复杂配置编辑。

Page Form：

- 用于添加 Provider、编辑 Provider、配置路由策略、配置复杂采集策略。
- 页面顶部有返回，底部有固定动作栏。

MetricPanel：

- 用于首页核心指标和页面摘要。
- 指标数量有限，优先 4 个核心指标。
- 避免随意新增一组视觉不同的指标卡。

## 分阶段落地

### Phase 1：设计系统与 Shell

- 建立颜色 token：白底、系统灰、Apple 蓝、状态色。
- 升级 Button、Card、Tabs、Dialog 基础语义。
- 把 AppShell 改成窄工具栏。
- 保留当前页面路由，不做业务数据重构。

### Phase 2：首页工作台

- 将总览改为代理工作台。
- 实现当前路由 + 今日指标双核心区。
- 实现轻量路由/Provider 队列。
- 保留本地代理入口复制能力。

### Phase 3：对象列表统一

- 抽象 ObjectRow。
- 中转站列表、Key 池列表、采集任务列表、路由队列逐步迁移。
- 统一 hover actions、状态徽标、拖拽柄、指标布局。

### Phase 4：沉浸式添加 Provider

- 新增页面级 AddProviderPage。
- 支持 preset 选择和自动填充。
- 支持 Custom / OpenAI Compatible / Sub2API / NewAPI。
- 将复杂配置从小弹窗迁移到页面表单。

### Phase 5：验收与收口

- 浏览器逐页验收：工作台、中转站、Key 池、采集、设置。
- 确认所有页面使用统一按钮、卡片、tabs、dialog 语义。
- 清理旧的青色背景和后台式卡片墙。

## 验收标准

- 页面背景和主要容器不再出现青色主题底。
- 侧栏默认是窄工具栏，页面不再像传统后台。
- 首页指标显著，但首屏仍能看到当前路由和对象队列。
- Provider/Key/采集任务/Route 对象行视觉一致。
- 添加 Provider 是页面级沉浸式流程。
- 每页最多一个蓝色 primary 主按钮。
- 复杂表单不再塞进小弹窗。
- TypeScript 和 Vite 构建通过。
- 关键页面经过浏览器视觉验收。

## 不做事项

- 不复制 cc-switch 的代码或业务实现。
- 不引入账号、云同步、插件市场等额外产品边界。
- 不把 Relay Pool Desktop 改成完整 cc-switch 替代品。
- 不在本轮 UI 升级中改变代理核心语义。
