# Relay Pool 命令权限与数据恢复可靠性设计

## 状态

- 设计日期：2026-07-16
- 设计状态：待用户审阅
- 适用范围：`v0.2.7` 之后的命令权限修复、数据目录启动可靠性、故障诊断和权威数据库选择
- 实施原则：分阶段发布；先消除确定性 ACL 故障，再处理需要历史状态才能触发的数据目录故障

## 背景与证据边界

`v0.2.7` 发布后，有用户反馈两个严重症状：

1. 进入“价格 / 倍率”页面时收到 `Command load_pricing_comparison_workspace not allowed by ACL`。
2. 更新并重启后，Relay Pool 内的站点列表为空，用户认为需要重新导入或重新添加。

第一个症状已经有确定证据：页面切换性能改造新增了 `load_channel_status_workspace`、`load_pricing_comparison_workspace` 和 `mark_change_events_read` 三个 Tauri 命令，并把它们注册进 `generate_handler!`，但没有同步加入 `src-tauri/permissions/main-window.toml`。现有 ACL 一致性测试可以准确发现遗漏，但该测试没有接入标准构建和 Release 门禁。

第二个症状发生在其他用户机器，当前没有取得故障机器的数据库路径、配置和 SQLite 检查结果。本设计采用以下审慎结论：

- 更新程序没有已知的主动删除数据库逻辑，数据库文件大概率仍在某个历史目录，但这只是待验证假设。
- 现有启动流程可能在配置目录中的数据库缺失或选择了另一份数据库时，由 SQLite 静默创建并打开一个新空库，形成“数据被清空”的观感。
- 历史自定义数据目录、默认目录和待激活目录可能各自保留不同数据库；不能根据文件大小、站点数量或修改时间自动认定其中一份一定正确。
- 在没有故障机器证据时，不允许设计静默跨库合并或自动覆盖。
- 当前产品还存在用户主动从设置页发起的数据目录搬迁：旧配置通过 `pendingDataDir` 与 `sourceDataDir` 表达搬迁意图，启动时可能把源数据库复制到新目录。它与“从现有候选中找回权威库”是两条不同业务路径，必须分别建模和测试。

## 根本原因

### 命令权限根因

Tauri 命令能力分散在四个需要手动同步的表面：

1. Rust `#[tauri::command]` 定义。
2. `tauri::generate_handler!` 注册清单。
3. `main-window.toml` 窗口权限清单。
4. 前端 `invoke`、错误分类和旧后端兼容回退。

本次改造更新了第 1、2、4 层，遗漏第 3 层。同时出现了三项流程失效：

- 实施计划没有把 ACL 文件列为新增命令的必改表面。
- fallback 测试使用了猜测的 `not allowed. Command not found`，没有覆盖发布包实际返回的 `not allowed by ACL`。
- 已有的注册命令与 ACL 全量一致性测试没有进入 `pnpm build` 或 Release 工作流。

因此根因不是单一漏项，而是跨层不变量没有成为强制发布条件。

### 数据可见性根因

当前数据库启动逻辑同时承担了目录解析、数据库选择、目标目录创建、空库创建、schema 初始化和数据目录激活。它没有显式区分三种语义：

- 真正第一次启动，可以创建新数据库。
- 用户明确发起的数据目录切换，可以准备新目标。
- 已初始化安装再次启动，预期数据库缺失或不可访问，此时必须阻止启动并请求恢复。

当第三种情况被当作第一种处理时，应用会正常打开空库，页面只能看到“还没有中转站”。即使旧文件仍在，用户也无法知道应用打开了哪一份库。

当前实现还存在三项已经能从代码确认、但不等于故障机器精确触发链的风险：

- `AppDatabase::initialize` 先创建所选目录，再准备/复制数据库，最后才调用 `Connection::open`；因此“目录存在”不能证明数据库曾成功打开。
- 待切换目录可能在数据库正式打开和 schema 初始化之前被 `mark_data_dir_initialized` 提交；后续打开或迁移失败时，配置中的 source 线索可能已经被改写。
- 现有搬迁只用站点数量和少量设置判断“是否有用户状态”，并通过裸 `fs::copy` 复制 SQLite 主文件；该判定不覆盖所有业务表，复制也没有对 WAL 一致性提供保证。

这也解释了为什么开发机器可能一直正常而其他用户更新后出错：只有带历史 `pendingDataDir/sourceDataDir`、自定义目录、空目标库、WAL 状态或路径权限差异的机器才会进入这些分支。当前机器正常只能证明当前机器没有触发相同状态组合，不能反证发布包可靠。

## 设计目标

1. 主窗口调用的已注册命令必须全部获得对应 ACL；漏项不得进入构建产物。
2. ACL 拒绝必须作为发布配置故障暴露，不能被旧版本兼容回退吞掉。
3. 已初始化安装在预期数据库缺失、不可访问、损坏或选择冲突时，不得静默创建空库。
4. 数据恢复必须以完整 SQLite 数据库为原子单位，不进行站点、Key、设置或 secret 的行级跨库拼接。
5. 数据库选择必须可预览并由用户确认；显式选择现有数据库时必须先创建一致性整库备份，再提交活动目录配置。迁移仍由下一次冷启动的唯一正常初始化路径执行，失败时必须 fail-closed。
6. 代理、采集器和渠道监控运行器只能在活动数据库确定且验证成功后启动。
7. 能够导出不包含敏感信息的诊断报告，以支持远程用户故障定位。
8. 修复不得削弱授权捕获窗口的最小权限边界。

## 非目标

- 不自动合并两份都包含用户状态的数据库。
- 不根据“站点更多”“文件更大”或“修改时间更新”静默选择权威库。
- 不在本轮实现通用数据库编辑器、任意表合并或云备份。
- 不在恢复入口中重新实现或泛化“移动数据库”；设置页的数据目录搬迁保留为独立能力和独立实现计划，不得偷偷复用恢复选择流程。
- 不把 SQLite、API Key、Cookie、Local Key、secret 密文或完整请求日志上传到远端。
- 不借本次修复调整页面布局、路由算法、采集语义或价格计算规则。

## 可靠性不变量

以下不变量必须由自动化测试约束：

1. **命令授权完整性**：除明确限制给其他窗口的命令外，主窗口 `generate_handler!` 中的命令必须全部存在于 `main-window` 权限清单。
2. **错误不降级**：`not allowed by ACL`、数据库错误和业务错误不得触发浏览器 mock 或旧命令回退。
3. **空库创建边界**：只有已确认的首次启动或用户明确发起的新目录准备流程可以创建数据库。
4. **活动库单一性**：一次进程生命周期只能有一个权威活动数据库。
5. **配置后提交**：活动目录配置只能在所选数据库只读检查、一致性备份和只读验证全部成功后提交；配置提交后通过冷启动完成正常打开和迁移。
6. **冲突不裁决**：两份候选数据库都有用户状态时必须进入冲突状态，由用户选择，不得自动覆盖。
7. **整库选择**：选择权威数据库时保留站点、Key、凭据、secret、分组、价格、余额、监控、采集、日志、变更事件和设置之间的内部一致性。
8. **先备份后激活**：显式选择现有数据库时必须先创建并验证一致性整库备份；正常打开已提交活动库不因每次启动重复制造备份。
9. **运行器后启动**：数据库选择完成前，Proxy、Collector 和 Monitor 不得读取或写入任何候选库。
10. **空态可信**：UI 只有在数据库读取成功且站点数确实为零时才能显示“还没有中转站”；打开失败或恢复待处理必须显示错误状态。

## 总体方案

实施拆成三个内部阶段，每个阶段独立验证和提交。对外只发布两个版本：阶段一形成紧急 ACL 热修；阶段二和阶段三必须合并为一个完整的数据可靠性版本，不能发布“只会阻止启动、却不能选择现有数据库”的中间产物。

### 阶段一：ACL 热修与发布门禁

阶段一不修改数据库选择或持久化行为。

#### 命令授权

把以下命令加入 `main-window`：

- `load_channel_status_workspace`
- `load_pricing_comparison_workspace`
- `mark_change_events_read`

`record_capture_event` 等捕获窗口专属命令继续只属于 capture capability；不得因为全量校验而扩大到主窗口或其他窗口。

#### 错误分类

前端 IPC 错误分为三类：

| 类型 | 示例 | 行为 |
|---|---|---|
| Runtime 不存在 | 浏览器预览没有 `__TAURI_INTERNALS__` | 允许使用浏览器预览 mock |
| 旧后端命令不存在 | `Command ... not found` | 仅对明确支持旧后端的读取或批量 API 使用旧命令回退 |
| ACL / 数据库 / 业务错误 | `not allowed by ACL`、SQLite 错误 | 直接失败并显示真实错误，不回退 |

兼容回退的目的只是支持开发预览或短期前后端版本差异，不得成为掩盖发布配置错误的兜底。

#### 命令契约门禁

新增统一契约测试入口 `test:contracts`，由一个跨平台 Node runner 执行仓库中明确登记的契约测试。至少包括：

- `generate_handler!` 与 `main-window.toml` 的全量包含关系。
- capture capability 的精确最小权限。
- 前端关键 `invoke` 命令已注册。
- 命令不存在、Runtime 不存在和 ACL 拒绝三类真实错误文本。

统一 CI 入口必须依次执行：

1. `test:contracts`
2. Vitest
3. TypeScript / Vite build
4. Rust focused tests 与 `cargo check`

Release 工作流不得自行维护一份零散的测试子集，而应调用同一个 CI 入口。Tag 发布只能在该入口成功后继续生成安装包。

### 阶段二：启动诊断与 fail-closed 数据目录

阶段二解决“文件可能仍在，但应用打开了错误数据库或新空库”的风险，不提供自动合并。

#### 安装状态与配置版本

数据目录配置升级为带版本的原子配置，只保留启动选择真正需要的字段：

```json
{
  "version": 2,
  "activeDataDir": "当前已提交目录",
  "pendingDataDir": null,
  "sourceDataDir": null,
  "updatedAt": "时间戳"
}
```

配置通过临时文件、刷盘和同目录原子替换提交。配置中不保存 secret。

从新版本开始，在默认应用数据目录保存一个独立的非敏感 installation marker，只表达“这台安装至少成功初始化过一次”。marker 不写入业务数据库，不包含随机用户标识、设备标识、数据库标识或计数。它只用于区分 `FirstRun` 与“已初始化安装找不到预期数据库”；旧版本没有 marker 的历史歧义仍按本节兼容规则处理。除非后续出现“同一路径中的数据库被替换且无法通过候选检查发现”的已证实故障，本轮不向数据库 schema 增加 `databaseId`。

#### 旧配置兼容

现有 `relay-pool-data-dir.json` 只有 `pendingDataDir` 和 `sourceDataDir`。读取旧配置时：

- 先以只读方式检查默认、pending 和 source 候选；不得先重写配置。
- 只有一个候选数据库健康且包含用户状态时，可以把它作为建议活动库。
- 多个候选都有用户状态时进入冲突状态。
- 候选选择、一致性备份和只读验证成功后，才迁移为 v2 配置；下一次冷启动再执行唯一的正常数据库初始化路径。
- 旧配置迁移失败时保留原文件，不生成半迁移配置。

#### 启动候选检查

数据库正式打开和运行器启动前，对候选路径做只读检查：

- 路径是否存在且为普通文件。
- 只有在文件存在后才使用 SQLite `READ_ONLY` open flags 打开；检查代码不得调用会创建缺失文件的默认 `Connection::open`。
- SQLite header 是否有效。
- 文件大小和修改时间。
- `PRAGMA quick_check`。
- 可识别 schema/table 是否存在。
- 站点、Key、监控等非敏感计数。

检查不得读取或记录站点名称、URL、凭据、Local Key、Cookie、请求正文或 secret 内容。

#### 启动决策表

| 场景 | 行为 |
|---|---|
| 首次启动，无配置且默认数据库不存在 | 创建默认数据库并写入 v2 活动配置 |
| 活动数据库存在且健康 | 正常打开 |
| 活动目录不可访问或数据库缺失 | 阻止数据层启动，进入恢复界面 |
| 活动数据库损坏 | 阻止启动，显示完整性错误与备份/选择入口 |
| 当前候选为空、另一个候选有用户状态 | 提示可能打开了错误数据库，要求用户选择 |
| 只有一个健康且有用户状态的旧配置候选 | 预选该候选，但在切换活动配置前仍要求确认 |
| 多个候选都有用户状态 | 显示冲突，不自动选择 |
| 已初始化安装的候选全部缺失 | 不创建空库；允许定位数据库或明确创建全新库 |
| 旧配置表示 `pendingDataDir != sourceDataDir` | 识别为待完成的数据目录搬迁，不当作普通首次启动或普通恢复；按下述兼容门禁处理 |

旧版本没有 installation marker 且没有任何配置/数据库时，无法区分真正首次启动和数据库被外部删除。兼容策略是把它视为首次启动；从新版本开始，v2 配置和默认应用数据目录中的 marker 消除该歧义。marker 存在但活动数据库缺失时必须进入 `NeedsRecovery`，不得创建空库。该历史边界必须在实现和文档中明确说明。

#### 启动顺序

当前 `src-tauri/src/lib.rs` 在 `AppDatabase::initialize` 失败时会直接让 Tauri setup 失败，因此仅把数据库打开改成 fail-closed 仍不足以提供恢复入口。Tauri setup 调整为：

1. 初始化 SecretManager。
2. 解析配置并只读检查候选数据库。
3. 得到只读启动决策，但暂不注册可变的全局状态。
4. 若结果为 `Ready` 或 `FirstRun`，尝试打开活动数据库、执行现有迁移链并验证；成功后注册不可变的 `DataStoreStartupState::Ready`、`AppDatabase` 和运行器。
5. 若只读决策为 `NeedsRecovery` / `Conflict`，或第 4 步打开、初始化、迁移失败，则注册对应的不可变 `DataStoreStartupState` 和脱敏候选摘要。setup 返回成功以保留主窗口，但不注册假的/空的 `AppDatabase`，也不启动任何运行器；前端只渲染恢复屏幕。
6. 用户完成选择后，后端只做只读验证、一致性备份和活动目录配置提交，然后调用现有进程重启能力受控重启应用；不在同一进程中迁移或动态替换数据库，也不补注册运行器。
7. 冷启动重新执行同一决策；只有得到 `Ready` 才渲染正常业务页面。

恢复屏幕只调用不依赖 `AppDatabase` 的最小诊断/选择命令。所有正常业务页面和其查询在启动状态为 `Ready` 前都不得挂载，因此不会触发缺失的 `State<AppDatabase>`。不得为兼容现有命令注入空库、内存库或 `Option<AppDatabase>` 并把错误降级成空集合。

### 阶段三：显式选择现有数据库

恢复界面只在启动诊断发现缺失、损坏、空库/旧库冲突时出现，也可从设置页手动进入只读诊断。

#### 候选展示

对每个候选只展示：

- 路径。
- 是否存在、是否可读。
- 文件大小和修改时间。
- 完整性检查结果。
- schema 兼容状态。
- 站点、Key、监控等计数。

修改时间、文件大小和计数只能帮助用户判断，不能被代码作为自动覆盖依据。

#### 恢复选择

用户可以：

1. 选择一份健康候选作为权威数据库。
2. 手动定位另一份 `relay-pool-desktop.sqlite3`。
3. 取消并退出。
4. 在二次确认后创建全新数据库；该操作不得删除或覆盖任何候选文件。

恢复不提供“保留 A 的站点、采用 B 的设置”等行级合并选项。选择某份数据库意味着整库采用其内部一致的站点、Key、secret、设置和历史数据。其他候选库保留在原位置，应用不得修改、删除或覆盖它们。

#### 选择与激活事务

恢复的本质是让配置指向用户选中的现有健康数据库目录，不是把数据库搬到另一个目录。流程如下：

1. 确认 Proxy、Collector 和 Monitor 未启动。
2. 以只读方式重新检查所选数据库的路径、header、`quick_check`、schema 和关键计数，避免使用已经变化的候选结果。
3. 确认所选目录满足数据库正常运行所需的读写权限。
4. 使用 SQLite backup API 或等价一致性快照，在默认应用数据目录的 `backups/<timestamp>/` 下创建固定文件名的整库备份，并验证备份可打开；不得对可能含 WAL 的数据库只做裸 `fs::copy`。备份不得覆盖源文件，也不得使用会被普通候选扫描误判为活动库的位置或名称。
5. 对原数据库和备份执行完整性、外键、schema、关键计数和必要的 SecretManager 解密验证；不得记录 secret 明文。原数据库与备份的关键计数必须一致。
6. 把该备份目录登记为恢复候选；候选发现只检查应用自有 `backups/` 的一级时间戳子目录，不递归遍历用户目录或磁盘。备份只读保留，不自动覆盖原数据库。
7. 验证成功后，原子写入 `activeDataDir`，清理已消费的 `pendingDataDir`，保留 `sourceDataDir` 仅用于兼容诊断。
8. 返回“配置已提交，需要重启”，由应用受控重启；运行器和业务页面只在下一次冷启动判定为 `Ready` 后启动。

第 2 至第 6 步失败时不得修改活动目录配置。第 7 步失败时，所选数据库和备份都保持不变，应用继续停留在恢复屏幕。受控重启调用失败时显示“请手动重启”，但不撤销已经验证并提交的活动库选择。下一次冷启动中的数据库打开或现有迁移链失败时进入 `NeedsRecovery { reason: OpenOrMigrationFailed }`，不启动运行器，并把迁移前备份列为候选；应用不得自动用备份覆盖原数据库。

本轮恢复入口明确不复制所选数据库到另一个活动数据目录，也不实现临时数据库替换、跨文件系统原子移动或恢复崩溃日志状态机。设置页既有的“移动数据”能力只能走下述独立搬迁合同，不能复用恢复入口偷偷搬迁文件。

整库备份与源数据库包含相同敏感数据，必须继承默认应用数据目录的本机访问限制，不进入日志、诊断报告、Git 或自动上传。第一版不做自动清理，避免误删唯一可用备份；恢复屏幕只提供“打开备份所在目录”，备份删除和保留策略留给独立设计。由于只在用户显式选择数据库时创建，正常启动不会持续堆积备份。

#### 现有数据目录搬迁兼容门禁

设置页主动发起的数据目录搬迁继续是受支持功能，但不属于本恢复流程。阶段二开始改动 `AppDatabase::initialize` 前，实施计划必须先把搬迁抽成独立任务，并保持以下合同：

- 搬迁意图只能由用户在设置页明确创建，不能由候选扫描自动生成。
- 源数据库始终保留不动；目标数据库已有任何可识别用户状态时不得覆盖。
- 新搬迁不得先创建空目标数据库。复制必须使用 SQLite 一致性备份到目标目录临时文件，验证通过后只在目标固定数据库文件不存在时同目录原子改名，然后提交活动目录配置。
- 对旧版本遗留的“目标已存在但看似为空”状态，不得依赖不完整的计数自动覆盖；进入 `NeedsRecovery { reason: PendingRelocation }`，让用户选择继续使用 source、手动选择 target，或在单独的搬迁流程中处理。
- 复制、验证或配置提交失败时，source 仍是权威库；重启后根据 source、target 和原配置重新决策，不维护另一套恢复日志状态机。

阶段二不得在搬迁兼容测试完成前发布。如果独立搬迁任务无法同时满足这些条件，应暂时禁用“更改数据目录”入口并保留旧活动库，而不是用恢复模块中的分支绕过门禁。

#### 数据库损坏

完整性检查失败的数据库不允许直接设为活动库。恢复界面可导出诊断并引导用户保留文件，不在本轮自动执行 SQLite 数据抢救或逐表重建。复杂损坏恢复需要独立工具和独立设计。

## 脱敏诊断报告

诊断报告是本地生成、由用户主动导出的 JSON 或文本文件，包含：

- 应用版本、bundle identifier 和操作系统版本。
- 配置版本、installation marker 是否存在、活动/pending/source 候选角色和每份候选的单次报告内匿名编号；不导出绝对路径、用户名或目录名。
- 候选文件存在性、大小、修改时间。
- schema 版本和关键表计数。
- `quick_check`、`integrity_check`、`foreign_key_check` 结论。
- 最近一次目录选择或迁移的阶段、结果和脱敏错误码。

诊断报告禁止包含：

- 站点名称、域名和 URL。
- API Key、Cookie、Token、Local Key。
- secret 密文、nonce、AAD 或 value hash。
- 请求路径中的敏感参数、请求/响应正文。
- 完整数据库文件或完整应用日志。

## 用户界面与错误语义

### ACL 错误

ACL 拒绝显示为明确的应用版本/权限配置错误，保留命令名用于诊断，但不展示调用参数。UI 不显示“暂无数据”。

### 数据库启动错误

正常 AppShell 之前提供一个最小恢复屏幕：

- 说明“数据文件没有被删除；应用当前无法安全确定应打开哪一份数据库”时，必须使用条件措辞，不能在未检查文件前保证文件存在。
- 展示候选摘要和推荐原因。
- 切换活动目录和创建全新库要求二次确认；只读诊断不要求确认。
- “创建全新库”明确说明不会删除旧文件，但会让新库成为活动库。

### 页面查询错误

数据库已正常启动后，页面查询失败必须显示错误面板并允许重试。只有成功返回空集合时才显示业务空态。

## 对现有功能的影响

| 功能 | 阶段一影响 | 阶段二/三影响与保护 |
|---|---|---|
| 价格页 | 恢复聚合工作区读取 | 活动库未确定时不进入页面 |
| 渠道状态 | 修复同类 ACL 潜伏错误 | Monitor 在数据库验证后启动 |
| 变更中心 | 批量已读真正持久化 | 旧后端仅在命令确实不存在时回退 |
| 中转站资产 | 无业务语义变化 | 查询失败不再伪装成空站点 |
| Key、凭据、Secret | 无变化 | 整库恢复，禁止行级拆分；恢复后做解密检查 |
| 本地代理 | 无变化 | 恢复期间不启动；成功后从权威库读取设置 |
| 采集器与监控 | 无变化 | 数据库确定后启动，避免写入错误候选 |
| 自定义数据目录 | 无变化 | 复用“检查、确认、提交”边界，但不引入通用状态机或隐式搬库 |
| 自动更新 | ACL 热修直接受益 | 必须覆盖上一正式版到候选版的重启测试 |
| 捕获授权窗口 | 权限保持不变 | 不参与恢复流程 |
| 浏览器预览 | Runtime 不存在时继续使用 mock | 不模拟真实数据库恢复 |

## 可扩展性与可维护性

### 命令面

- 命令注册与 ACL 保持现有 Tauri 结构，不引入绕过权限系统的总权限。
- 通过全量契约检查消除人工同步遗漏；新增命令时不需要记住额外测试名单。
- 错误分类集中维护，业务 API 不再各自用 `/invoke/i` 等宽泛正则判断。

### 数据目录面

- 后端只新增一个窄领域目录，不继续把启动选择塞入已承担正常 CRUD 与迁移职责的 `database.rs`：

  ```text
  src-tauri/src/services/data_store/
    mod.rs       启动编排和最小 DataStoreStartupState，不含 SQL 和 UI 文案
    config.rs    配置读取、原子写入和版本迁移
    inspect.rs   候选数据库只读检查
    decision.rs  纯启动决策函数
    backup.rs    仅负责显式选择前的一致性整库备份
  ```

- `database.rs` 继续只拥有 `AppDatabase`、现有 schema 初始化/迁移和正常 CRUD/query；本设计不得借机重写全部历史迁移。候选扫描、用户选择、配置提交或恢复 UI 决策不得加入该文件。
- `inspect.rs` 只能接收明确候选路径并以 `READ_ONLY` flags 打开 SQLite；不得遍历整块磁盘、创建目录/文件或执行 schema 初始化。
- 前端边界固定为 `src/features/data-recovery/DataRecoveryScreen.tsx`、`recoveryViewModel.ts`、`src/lib/api/dataRecovery.ts` 和 `src/lib/types/dataRecovery.ts`；业务页面不得各自保存恢复状态或复制候选判断。
- `DataStoreStartupState` 不是新的全局状态框架，只保存本次冷启动的不可变决策和候选摘要。恢复命令成功后只提交配置并重启，不支持运行中热切换 `AppDatabase`。
- 启动决策只使用 `Ready`、`FirstRun`、`NeedsRecovery { reason }`、`Conflict { candidates }` 四种结果。缺失、不可访问、损坏和迁移失败是 `NeedsRecovery` 的 reason code，不再扩张成互相重叠的顶层状态。`decision.rs` 必须是无 I/O 的纯函数并使用表驱动测试。
- 配置带版本，未来可迁移而不是依赖字段缺失推断。
- 整库选择避免为每张新业务表维护额外合并规则；未来增加表时仍保持内部一致。
- 不创建通用 recovery framework、不引入新的全局状态库、不增加行级合并 helper；ACL 错误分类和数据目录恢复不能出现在同一模块。
- IPC 错误文本分类只在一个现有公共错误模块维护，业务 API 不得复制正则。
- 新文件以单一职责和可测试性为硬边界，目标不超过约 300 行；接近上限时按上面的既定职责拆分，不通过 `utils.rs`、`manager.rs` 或布尔参数堆砌行为。
- 阶段一必须保持为独立小改动，不得为了“顺便统一架构”依赖阶段二/三的任何模块。

## 测试矩阵

### 命令与 ACL

- 每个主窗口注册命令都在 `main-window` 中。
- capture-only 命令不在主窗口权限中。
- 真实 `not allowed by ACL` 不触发 fallback。
- 真实 `Command ... not found` 仅对声明支持旧后端的 API 触发 fallback。
- 浏览器预览 Runtime 不存在才允许 mock。
- 价格、渠道和批量已读的发布包 IPC smoke test。

### 数据目录启动

- 全新安装，无任何文件。
- 默认目录已有健康数据库。
- 自定义活动目录已有健康数据库。
- 活动目录不存在、不可读或只读。
- 活动数据库缺失。
- 候选文件缺失时只读检查不创建文件或目录。
- 活动数据库 header 无效。
- `quick_check` 或 `integrity_check` 失败。
- 旧 v1 配置只有 pending/source。
- pending 目标为空、source 有站点。
- pending 目标只有设置、source 有站点。
- source 与 target 都有用户状态。
- 默认、source、pending 三份库同时存在。
- installation marker 存在但活动数据库缺失。
- 配置写入一半、临时配置残留。
- 数据库打开或 schema 初始化失败时，不得提前提交 `activeDataDir` 或丢失旧 `sourceDataDir`。
- `NeedsRecovery` 时 Tauri setup 成功、恢复屏幕可用、`AppDatabase` 与运行器均未注册。
- 恢复屏幕未挂载任何正常业务页面，也没有业务查询被发出。

### 选择与迁移

- 选择现有健康数据库后不复制到另一个活动目录；除独立一致性备份外，只更新活动目录配置。
- 正常打开已提交活动库时不创建重复备份。
- 显式选择 WAL 模式数据库时能在独立备份目录生成一致性整库备份。
- 备份包含完整敏感数据但不进入诊断导出、日志或自动上传；正常启动不创建备份。
- 备份前磁盘空间不足或备份创建/验证失败。
- foreign key 检查失败。
- secret 解密检查失败。
- 冷启动的现有迁移链失败时进入恢复屏幕、不启动运行器，并展示迁移前备份候选。
- 配置提交失败时原活动目录保持不变，迁移前备份保留。
- 配置提交后受控重启失败时提示手动重启，不重复提交或回滚活动目录。
- 冷启动中运行器启动失败时不切换回其他候选，并可重试启动。
- 未选中的候选数据库内容和时间戳保持不变。
- 恢复成功后站点、Key、设置、监控和日志计数保持一致。

### 设置页数据目录搬迁

- source 为 WAL 模式时生成一致性目标数据库，不能只复制主文件。
- target 不存在时可完成搬迁；target 已有任意可识别业务状态时拒绝覆盖。
- 旧版本遗留的空 target 进入 `PendingRelocation`，不依赖站点/设置计数自动覆盖。
- 复制或验证失败时 source 配置保持不变，target 临时文件不被当作活动库。
- 目标数据库验证成功但配置提交失败时，source 仍是活动库，重启后能重新决策。

### 升级验证

- 从上一正式安装版通过 updater 升级到候选版。
- 默认数据目录升级。
- 自定义数据目录升级。
- 带旧 v1 pending 配置升级。
- 更新前后记录活动数据目录和关键计数。
- 更新后依次打开价格、渠道、变更中心和站点页面。
- 完整退出并重新启动后再次验证。

## 发布策略

### 发布一：紧急 ACL 热修

- 只包含 ACL、错误分类和发布门禁修复。
- 不改数据库选择策略。
- 验证正式打包产物，而不只验证 Vite/开发模式。

### 数据可靠性版本的内部阶段二：诊断与 fail-closed

- 引入 v2 配置、候选检查、恢复状态和脱敏诊断。
- 检测到歧义时阻止业务运行，但不自动复制或替换数据库。
- 先通过内部构造的旧配置矩阵和上一正式版升级测试。

### 数据可靠性版本的内部阶段三：显式选择权威数据库

- 在阶段二稳定后开放候选选择、必要的迁移前备份、验证和活动目录提交。
- 选择功能必须通过备份失败、配置提交失败、受控重启失败和冷启动打开/迁移失败测试；不包含数据库复制到活动目录或替换协议。

阶段一与数据可靠性版本分别提交、分别发布并保留独立回滚点。阶段二/三可以分提交开发和验证，但必须在同一个候选版本中完成端到端升级测试后一起发布；不把 ACL 紧急修复与数据库启动改造一次性交付。

## 验收标准

1. 当前 ACL 一致性测试从明确 RED 变为 GREEN，并进入本地构建、CI 和 Release 三条强制链。
2. 正式打包应用可打开价格和渠道页面，批量已读可持久化，且 capture 权限没有扩大。
3. 已初始化安装在活动数据库缺失或冲突时不创建空库、不启动运行器、不显示业务空态。
4. 旧配置迁移不会在候选验证成功前重写原配置。
5. 两份有用户状态的数据库永不自动合并或覆盖。
6. 显式选择现有数据库前存在可打开且关键计数一致的整库备份；后续打开或迁移失败时不启动运行器，并允许用户选择该备份。
7. 选择完成后通过完整性、外键、schema、secret 和关键计数验证，未选中的数据库未被修改。
8. 脱敏诊断报告不包含任何可还原凭据或用户业务详情的信息。
9. 从上一正式版完成真实 updater 升级、完整退出和冷启动验证。
10. 设置页数据目录搬迁在 WAL、目标冲突、复制失败和配置提交失败场景下都不覆盖 source，且阶段二/三不会以不可自救的中间版本单独发布。

## 自审结论

本设计把确定性的 ACL 回归和尚需故障机器证据的数据目录问题分开交付，避免用高风险数据库改造拖延紧急修复。恢复策略采用整库选择而非行级合并，能随着业务表增加继续保持内部一致性；冲突时由用户确认，避免站点数量或文件时间等启发式规则造成永久覆盖。

设计明确了首次启动与已初始化启动的不同空库创建权限、配置后提交、运行器后启动、显式选择前整库备份、旧配置兼容和敏感信息边界。恢复只选择现有目录，不复制到活动目录、不替换或合并数据库；备份只作为独立候选保留，因而不需要跨文件系统原子替换和第二套崩溃恢复状态机。本设计也不借机重写全部历史 schema migration。文档没有遗留占位项、自动裁决、隐式覆盖或通用恢复框架。唯一不可消除的历史边界是：旧版本既没有配置又没有数据库时，无法区分首次启动与外部删除；该边界已明确记录，并由新版本的 installation marker 和 v2 配置向后消除。
